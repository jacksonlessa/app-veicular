# Task 1.0 Review — Bootstrap Next.js + TypeScript + ESLint + Prettier

## Verdict: CHANGES REQUIRED

## Findings

### Critical

**C1 — Estrutura Clean DDD ausente**
A task define (subtarefa implícita da TechSpec, passo 3 do sequenciamento) que os diretórios `src/domain/`, `src/application/`, `src/infrastructure/`, `src/components/ui/` e `src/lib/` devem ser criados com `.gitkeep`. Nenhum desses diretórios existe. O `src/` contém apenas `src/app/`. Embora a criação formal dos diretórios esteja na TechSpec e o PRD exige RF-2.1, a task 1.0 é o único momento de bootstrap — omitir a estrutura impede todas as tarefas subsequentes de seguir a arquitetura acordada sem retrabalho.

**C2 — Dependências obrigatórias não instaladas**
`package.json` contém apenas `next`, `react` e `react-dom` nas dependências e não inclui nenhuma das libs mandadas pela TechSpec:
- `@prisma/client`, `next-auth`, `clsx`, `tailwind-merge`, `class-variance-authority`, `lucide-react` (dependencies)
- `prisma` (devDependencies)

Estas dependências são pré-requisito para todas as tasks subsequentes (Prisma na task 2, NextAuth na task 3, shadcn/cn() na task 5).

### High

**H1 — Tokens do tema Âmbar não aplicados no `globals.css`**
O arquivo `src/app/globals.css` contém o CSS gerado pelo `create-next-app` com variáveis de Geist (`--background`, `--foreground`, dark mode via `prefers-color-scheme`) em vez dos tokens Âmbar definidos na TechSpec (`--color-bg: #F0EEE8`, `--color-accent: oklch(0.58 0.19 38)`, etc.). O bloco `@theme` necessário para o Tailwind v4 consumir os tokens como classes utilitárias está ausente. Isso invalida o critério de aceitação 4 da TechSpec ("página `/` renderiza com fundo Âmbar").

**H2 — Fonte Plus Jakarta Sans não configurada**
`layout.tsx` carrega Geist Sans e Geist Mono via `next/font/google` e aplica `--font-geist-sans`/`--font-geist-mono`. A fonte exigida pela TechSpec (RF-5.3) é **Plus Jakarta Sans** com pesos 400, 500, 600, 700 e 800. O `html` tag também usa `lang="en"` em vez de `lang="pt-BR"` como especificado na TechSpec.

**H3 — `page.tsx` é o template padrão do `create-next-app`**
O conteúdo de `src/app/page.tsx` é o boilerplate gerado automaticamente (logo Next.js, links para docs, Deploy to Vercel). A TechSpec especifica que a página de smoke test deve renderizar o logo **RodagemApp** e um `<Button>` shadcn com cor Âmbar. O arquivo deve ser reescrito conforme o passo 10 do sequenciamento.

**H4 — `prisma/` directory e schema ausentes**
Não existe `prisma/schema.prisma` nem migration inicial. A TechSpec descreve o schema completo com 7 models (Account, User, Invite, Vehicle, Fuelup, Maintenance, MaintenanceItem). Embora a task 1.0 não liste explicitamente Prisma entre seus requirements, o TechSpec consolida a Fase 0 como uma única fase e o PRD cobre RF-3.1 a RF-3.3 como entregáveis desta fase. A ausência de Prisma bloqueia a task subsequente de banco de dados.

**H5 — `metadata` não atualizada**
`layout.tsx` mantém `title: "Create Next App"` e `description: "Generated by create next app"`. Deve ser renomeado para refletir o produto (RodagemApp).

### Medium

**M1 — `.env.example` ausente**
O PRD exige RF-6.1: `.env.example` com `DATABASE_URL`, `NEXTAUTH_SECRET` e `NEXTAUTH_URL`. O arquivo não foi criado. `.env.local` também não existe (aceitável, pois é ignorado pelo git), mas `.env.example` deve ser versionado.

**M2 — `.gitignore` não inclui `prisma/dev.db*`**
O `.gitignore` atual usa `".env*"` que cobre arquivos de ambiente, mas não inclui `prisma/dev.db` e `prisma/dev.db-journal`. A TechSpec especifica explicitamente que esses arquivos devem estar no `.gitignore`.

**M3 — README não documentado para o projeto**
O README gerado pelo `create-next-app` documenta o projeto genérico Next.js. O PRD e a TechSpec exigem que o README documente os passos de setup do RodagemApp: `clone → install → cp .env.example .env.local → npx prisma migrate dev → npm run dev`.

**M4 — NextAuth scaffolding ausente**
`src/app/api/auth/[...nextauth]/route.ts` não foi criado. RF-4.1 e RF-4.2 do PRD são entregáveis da Fase 0.

**M5 — `tsconfig.json` sem `baseUrl`**
A TechSpec especifica `"baseUrl": "."` nos `compilerOptions`. O `tsconfig.json` gerado contém `paths` mas não `baseUrl`. Embora o Next.js com `moduleResolution: "bundler"` torne `baseUrl` opcional para os paths funcionarem, a ausência diverge da spec documentada.

### Low

**L1 — `shadcn/ui` não inicializado**
RF-5.2 do PRD lista 9 componentes shadcn a serem adicionados (`button`, `input`, `select`, `card`, `badge`, `tabs`, `switch`, `label`, `separator`). Nenhum deles está presente. `clsx`, `tailwind-merge` e `class-variance-authority` também não estão instalados.

**L2 — `src/lib/utils.ts` ausente**
A TechSpec define `src/lib/utils.ts` com a função `cn()` como parte da estrutura do projeto. O arquivo não foi criado.

**L3 — `next-env.d.ts` está no `.gitignore`**
O `.gitignore` gerado inclui `next-env.d.ts`. O arquivo `next-env.d.ts` é gerado automaticamente pelo Next.js e deve ser versionado (conforme recomendação oficial do Next.js e do próprio template). Está presente no repositório mas também listado no `.gitignore`, gerando inconsistência.

**L4 — `postcss.config.mjs` usa configuração Tailwind v3**
O `postcss.config.mjs` gerado pelo `create-next-app` referencia `@tailwindcss/postcss` que é correto para v4, mas deve ser verificado se não há configurações residuais de v3.

## Summary

A implementação realiza corretamente os itens mais básicos da task 1.0: Next.js 15 com App Router e TypeScript strict foi instalado, `.nvmrc` com `20` está presente, `.prettierrc` com as configurações corretas foi criado, o script `format` está no `package.json`, e `npm run lint` e `npm run build` passam. O ESLint com `next/core-web-vitals` está configurado.

No entanto, a implementação entregou apenas o output bruto do `create-next-app` sem aplicar as customizações obrigatórias definidas na TechSpec e no PRD. A maior parte dos entregáveis da Fase 0 — estrutura de pastas Clean DDD, tokens Âmbar, fonte Plus Jakarta Sans, Prisma, NextAuth, shadcn/ui, `.env.example` e README do projeto — está ausente. Esses itens não são todos estritamente requirements da task 1.0 (que foca em bootstrap + ferramentas), mas os findings C1 e C2 representam omissões que a task 1.0 deveria ter coberto como parte do sequenciamento da TechSpec, ou que precisam ser explicitados nas tasks subsequentes para não ficarem sem dono.

Os findings H1, H2, H3 são bloqueadores diretos do critério de aceitação da task ("smoke test visual do tema Âmbar"), mesmo que a task não liste explicitamente esses itens, pois a TechSpec os trata como parte do bootstrap da Fase 0.
